# AI 短剧扒剧与仿写程序 — 设计文档

日期：2026-07-04
状态：已与用户确认

## 1. 目标

一个本地 Web 程序，输入一部短剧的全部分集视频文件（几十到上百集），通过 Gemini 多模态模型逐集「扒剧」生成原剧完整剧本，再基于结构级仿写生成一部题材全新、但剧情结构/钩子节奏与原剧对齐的新短剧剧本。

## 2. 需求要点（已确认）

- **输入**：本地视频文件夹，一部剧的全部分集（如 `第01集.mp4` ~ `第98集.mp4`），按文件名自然排序确定集数，界面可手动调整
- **视频理解**：仅使用 **Gemini 原生视频通道**（Files API 上传视频直接分析），不做抽帧降级方案
- **文本环节**（拆解/设定/大纲/剧本生成）：LLM 可配置多家（Gemini / DeepSeek / GPT / Claude 等），统一适配层
- **仿写程度**：结构级仿写 —— 保留原剧剧情结构、钩子节奏、反转点，替换人物、背景、题材
- **产出**：新短剧剧本，集数与原剧一致，逐集对应原剧结构位
- **工作流**：分阶段人工确认，每阶段产物可编辑、可重新生成
- **界面**：本地 Web（Vue 3 + Vite 前端，FastAPI 后端）

## 3. 五阶段流水线

| 阶段 | 输入 | 产物 | 模型 |
|------|------|------|------|
| ① 逐集扒剧 | 视频文件 | 每集原剧剧本（标准格式）+ 结构标注（钩子/反转/爽点及场号） | Gemini（视频） |
| ② 全剧拆解 | 全部原剧剧本 | 拆解报告：人物关系、主线/支线结构、逐集节奏表、题材套路总结 | 可配置 LLM |
| ③ 新剧设定 | 拆解报告 + 人工输入 | 新题材、世界观、人物表、新旧人物映射表 | 可配置 LLM（AI 辅助） |
| ④ 新剧大纲 | 拆解报告 + 新设定 | 逐集大纲，每集标注对应原剧集的结构位 | 可配置 LLM |
| ⑤ 逐集剧本 | 对应原集剧本 + 该集大纲 + 设定 + 前一集新剧本结尾 | 每集新剧本（结构节奏对齐原剧，内容全换） | 可配置 LLM |

每阶段产物落盘为纯文本文件，可用任意编辑器修改，界面提供「基于当前版本继续」。

## 4. 剧本标准格式

采用用户提供的参考格式（原剧剧本与新剧剧本统一使用）：

- 场次头：`集-场号  时间  内/外  地点`（如 `1-4  夜  内  博物馆柜台`）
- `出场人物：` 行
- `▲` 开头 = 画面/动作描述
- `【字幕：…】`、`【特效：…】` = 屏幕字幕与特效标注
- 台词：`角色(语气/vo/os)：台词`（vo=旁白配音，os=内心独白）
- 全剧汇总导出：标题 + 基本信息（总集数/成功/失败/生成时间）+ 目录 + 逐集剧本，txt/md 格式

## 5. 架构

```
浏览器（Vue 3 + Vite 单页应用）
  项目列表 → ①扒剧 → ②拆解报告 → ③新剧设定 → ④大纲 → ⑤新剧剧本
        │ HTTP / SSE（进度推送）
FastAPI 后端
  ├─ 项目管理（一部剧 = 一个项目）
  ├─ 任务引擎（后台队列、断点续跑、暂停/继续、状态恢复）
  ├─ Gemini 视频分析器（Files API 上传 + 分析）
  └─ LLM 适配层（文本环节多家可切换）
存储
  ├─ SQLite：项目、集、任务状态索引
  └─ 文件系统：所有内容产物（剧本/报告/设定/大纲）
```

## 6. Gemini 扒剧流程（每集）

1. `ffprobe` 读取视频元信息校验（本机需装 ffmpeg，不转码）
2. Gemini Files API 上传视频 → 轮询等待 `ACTIVE`
3. 发送扒剧 prompt（内置剧本格式规范 + 少量示例），要求输出完整剧本 + 结构标注
4. 校验输出（场号连续性、格式合规）→ 落盘缓存；删除远端已上传文件
5. 失败自动重试 2 次；仍失败标记为失败集，界面可单集重跑

- **断点续跑**：每集完成即缓存；中断后重跑仅处理未完成的集
- **并发控制**：可配置并发数（默认 3），避免 API 限流

## 7. 数据目录

```
data/projects/<项目ID>/
├── project.json          # 项目配置：视频目录路径、模型配置、状态
├── episodes/
│   ├── ep001.script.md   # 第1集原剧剧本（可人工编辑）
│   ├── ep001.meta.json   # 结构标注 + 处理状态/耗时/token
│   └── ...
├── analysis/report.md    # ②全剧拆解报告
├── settings/new_drama.md # ③新剧设定（含人物映射表）
├── outline/outline.md    # ④逐集大纲
└── scripts/
    ├── ep001.md          # ⑤新剧第1集剧本
    └── ...
```

全局配置 `config.yaml`：Gemini API key、各家文本模型 key/base_url/model、并发数、重试次数。

## 8. 长上下文策略（阶段②）

上百集剧本可能超出单次上下文：先每 20 集生成「分段摘要」，再汇总生成全剧拆解报告；集数少时直接一步生成。

## 9. Web 界面交互

- **页签①扒剧**：填写视频文件夹路径 → 扫描列出分集（可手动调整集数对应）→ 开始扒剧 → 每集状态实时进度（SSE）→ 查看/编辑/单集重跑 → 导出全剧汇总
- **页签②拆解报告**：生成 → Markdown 展示 → 可编辑/重新生成
- **页签③新剧设定**：AI 建议 2-3 个新题材方向 → 用户选择或自写 → 表单+自由文本（题材/世界观/人物映射表）→「AI 完善设定」补全
- **页签④大纲**：生成逐集大纲（标注对应原剧结构位）→ 单集可编辑/单集重新生成
- **页签⑤新剧剧本**：批量或单集生成 → 查看/编辑/重生成，支持附加指令重生成（如"台词更口语化"）→ 导出全剧汇总

## 10. 错误处理

- 所有 LLM 调用失败指数退避自动重试
- 任务可随时暂停/继续；程序重启后从 SQLite 恢复任务状态
- 扒剧输出格式校验失败视为该集失败，走重试

## 11. 测试策略

- pytest 单测覆盖核心逻辑：剧本格式解析/校验、分段摘要切分、任务状态机；LLM 调用全部 mock
- Gemini 视频通道用单集真实短视频做冒烟测试

## 12. 技术栈

- 后端：Python 3.11+、FastAPI、SQLite（aiosqlite 或 SQLAlchemy）、google-genai SDK、httpx
- 前端：Vue 3 + Vite，Markdown 渲染，SSE 进度
- 依赖工具：ffmpeg（ffprobe）
